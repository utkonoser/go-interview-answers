# 02. Wildberries — дефицит на складе и проектирование остатков

**Контекст:** e-commerce / fulfillment (WB, Lamoda, Ozon FBO). На Middle+ часто спрашивают и **схему** (reserve/release), и **операционный SQL** под отчёт склада.

**Время:** ~30 мин (проектирование + 2 запроса).

## Дано

Базовая схема — `schema.sql`, данные — `seed.sql`.

Упрощения:

- один SKU = один `product` (без размерной сетки);
- заказ в статусе `pending` / `confirmed` / `shipped` / `cancelled`;
- `order_items` в `pending` и `confirmed` **резервируют** товар со склада.

## Часть 1. Проектирование

Дополни схему так, чтобы:

1. Хранить **остаток** и **зарезервированное** количество по паре `(warehouse_id, product_id)`.
2. Операции **reserve** и **release** были атомарны (опиши SQL или псевдокод в транзакции).
3. Нельзя зарезервировать больше, чем `available - reserved`.

Нарисуй таблицы и ключи в `design.md` (или на бумаге). Эталон — `design.md` в этой папке.

**Вопросы интервьюера:**

- зачем не хранить только `available` и не вычитать `SUM(order_items)` на лету?
- какие индексы на `order_items`?
- что при отмене заказа (`cancelled`)?

## Часть 2. Запросы

### 2.1. Дефицит

SKU, по которым **сумма qty** в позициях заказов со статусом `pending` или `confirmed` **строго больше** свободного остатка на **любом** складе, где лежит товар.

Свободный остаток = `warehouse_stock.quantity - warehouse_stock.reserved`.

Колонки: `sku`, `product_name`, `warehouse_name`, `free_qty`, `demand_qty`.

### 2.2. Отчёт по складам за 7 дней

По каждому складу за последние 7 дней (as-of `2025-06-15`): число **неотменённых** заказов и выручка в копейках (сумма `qty * price_kopecks` по позициям).

Исключить заказы со статусом `cancelled`.  
Колонки: `warehouse_name`, `orders_count`, `revenue_kopecks`.

## На что смотрят

- `CHECK`, `UNIQUE (warehouse_id, product_id)`, транзакции
- `JOIN` + `GROUP BY` + `HAVING`
- понимание разницы между агрегатом по заказам и снимком остатков
