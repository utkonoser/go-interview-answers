# 03. 2ГИС — бронирование номеров и занятость

**Контекст:** travel / booking в экосистеме 2ГИС (отели, POI, бронь номера). На Application Review и SQL-секции спрашивают про **пересечение дат**, отмены и отчёты по загрузке.

**Время:** ~30 мин (проектирование + 2 запроса).

## Дано

Схема — `schema.sql`, данные — `seed.sql`.

- `places` — объект (отель) в городе
- `rooms` — номер в отеле
- `bookings` — бронь с `check_in` / `check_out` (полуинтервал: ночь с `check_in`, выезд в `check_out`), статус `confirmed` / `cancelled`

Даты **не включают** `check_out` как ночь (как в отелях: 10–12 июня = 2 ночи).

## Часть 1. Проектирование

Как гарантировать на уровне БД, что у одного `room_id` **нет пересекающихся** броней со статусом `confirmed`?

Опиши в `design.md`:

- constraint / индекс (в PostgreSQL — `EXCLUDE` + `daterange`);
- что делать при отмене;
- нужен ли отдельный слот `room_nights` вместо интервала.

## Часть 2. Запросы

As-of дата: `2025-06-15`.

### 3.1. Загрузка отелей

На период **с 2025-06-15 по 2025-06-21** (7 ночей): для каждого отеля — `place_name`, `city`, `room_count`, `booked_nights`, `occupancy_pct` (забронированные ночи / (комнаты × 7) × 100, округление до 1 знака).

Учитывать только `confirmed`. Отменённые игнорировать.

### 3.2. Конфликты броней

Пары `booking_id_a`, `booking_id_b`, `room_id` для **confirmed** броней одного номера с пересекающимися датами (для аудита данных).

### 3.3. Свободные номера в городе

В городе **Сочи** на даты заезда **2025-07-01** — **2025-07-05** (4 ночи): номера, у которых **нет** пересечения с confirmed-бронями.

Колонки: `room_id`, `room_name`, `place_name`.

## На что смотрят

- `daterange`, `&&`, `OVERLAPS`, `generate_series` по дням
- агрегация «ночей» из интервалов
- осознанные индексы `(room_id, check_in)` и partial index `WHERE status = 'confirmed'`
