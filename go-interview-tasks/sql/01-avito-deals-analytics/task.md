# 01. Avito — аналитика объявлений и сделок

**Контекст:** маркетплейс объявлений (Avito, часть сценариев Ozon Seller). На собесе дают упрощённую схему и просят написать SQL без ORM.

**Время:** ~25 мин (запросы + обсуждение индексов).

## Дано

Схема и тестовые данные — `schema.sql`, `seed.sql`.

- `users` — продавцы и покупатели
- `listings` — объявления (`status`: `active`, `sold`, `archived`)
- `listing_views` — факт просмотра (один пользователь может смотреть много раз)
- `deals` — успешно завершённые сделки (цена фиксируется в момент сделки)

Суммы в **копейках** (`BIGINT`), как в продакшене.

## Часть 1. Запросы

### 1.1. Топ продавцов по выручке

Топ-5 продавцов по сумме `price_kopecks` завершённых сделок за **последние 30 дней** (от `CURRENT_DATE`) в городе **Москва**.

В результате: `seller_id`, `seller_name`, `deals_count`, `revenue_kopecks`.  
Сортировка: по убыванию выручки, при равенстве — по `seller_id`.

### 1.2. Объявления без конверсии

Объявления со статусом `active`, у которых **≥ 50 просмотров за всё время**, но **ни одной сделки** (ни когда-либо).

В результате: `listing_id`, `title`, `views_count`, `days_on_site` (дней с `created_at` до сегодня).

### 1.3. Воронка по дням

По каждому дню за последние 7 дней (включая сегодня): число **уникальных** объявлений с хотя бы одним просмотром и число завершённых сделок в этот день.

Колонки: `day`, `viewed_listings`, `deals_count`. Дни без активности — с нулями.

## Часть 2. Проектирование

Какие **индексы** добавишь для ленты:

```sql
SELECT *
FROM listings
WHERE city = $1 AND status = 'active'
ORDER BY created_at DESC
LIMIT 20;
```

Обоснуй порядок колонок в индексе и скажи, покроет ли он `ORDER BY` без sort.

## На что смотрят интервьюер

- `JOIN` vs подзапросы, `GROUP BY`, `HAVING`, `FILTER` / условная агрегация
- `COUNT(DISTINCT …)` для уникальных просмотров
- `generate_series` / календарь для дней без данных
- осознанный выбор индекса `(city, status, created_at DESC)`
